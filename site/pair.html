act 02 · bind

Point the name
at your real agent

Owning a name and operating an agent are different proofs, and PAN keeps them separate. Binding is the moment a name stops being a reservation and becomes a verified address: your agent proves, with its own key, that the name points at it.

the pairing moment

One signature turns a name into an address

For an agent that holds a key, and most personal-agent runtimes do, binding works like pairing a device. The registrar gives you a short code, your agent signs it, and the binding is the intersection of two proofs: the code proves you own the name, the signature proves you run the agent.

you
ask to pair Coder.jeff@gmail.com → code KX4-92F

In your session, request a pairing code. It is single-use and expires in ten minutes.

agent
sign pan-pair-v1:KX4-92F:<agent-id>

Whatever runs your agent, a gateway, a daemon, a five-line script, signs the code with the agent's key.

registrar
✓ bound · Coder.jeff@gmail.com → agent UD653KLV…

Signature verified against the agent's key. The name now resolves to that agent, logged with its method.

two ways to bind

Proof, matched to what you have

  • Agent-key pairing. The signature flow above. For any agent that holds a key, which is nearly all of them. This is the primary path.
  • Submitter-match. For an agent with no key of its own (say, an A2A card reachable only by URL), submit it under the same email that owns the name and it binds directly.
host neutrality is in the spec

The registrar may not require any particular agent host, framework, or network to pair. If a listing declares a key, whoever holds that key can bind, from any runtime. That is normative, not a promise.
