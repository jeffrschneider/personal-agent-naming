A draft standard, running

PAN: Personal Agent Naming.

PAN deliberately does one thing. It names agents, binds names to them, and defines what a resolved name returns. It does not carry messages, does not grade capabilities, and does not host anything. 

email tier:   PublicAgent.jeffrschneider@gmail.com   anchored to an inbox, notarized once
domain tier:  PublicAgent@jeffschneider.com          anchored to a domain, re-verifiable by anyone

Rule 1: nobody parses handles.

A handle is an opaque key. Resolution is exact-string lookup of the whole thing. No consumer decomposes it, ever, which is why the grammar stays this simple and the two tiers coexist without ceremony.

Rule 2: one string, one owner.

Uniqueness is full-string, first come first served, across both tiers. Second claimant hears "taken." Released handles sit in a 90-day cooling-off so a name you wrote down last year never silently starts pointing at a stranger.

The domain tier

Own a domain? Skip the notary.

Publish one file at https://yourdomain.com/.well-known/pan.json (or one DNS TXT record at _pan.yourdomain.com) naming your agents. That record is the claim, the proof, and the binding in one: the registrar mirrors it, re-checks it daily, and anyone on the internet can re-verify it without trusting anybody. If the record disappears, the card says stale instead of pretending. 

{
  "version": "pan-0.2",
  "handles": [
    { "name": "PublicAgent", "key": "UD653KLV…" }
  ]
}
Not just claimed. Bound.

Anyone can claim a name. Pairing proves it's your agent's.

Owning a name and operating an agent are different proofs, and PAN refuses to blur them. For agents that carry a key, binding works like pairing a Bluetooth device:

  1. You ask for a pairing code. The registrar issues a short single-use code, KX4-92F, good for ten minutes.
  2. Your agent signs one string. Whatever software holds the agent's private key (a gateway, a daemon, a five-line script) signs pan-pair-v1:KX4-92F:<agent-id>.
  3. The signature comes back. The code proves the handle owner started this; the signature proves agent control. The binding is the intersection of the two proofs, and it's logged with its method.
Host neutrality is normative in the spec, not a marketing promise. The registrar may not require any particular framework, network, or vendor for pairing. If a listing declares a key, whoever holds that key can pair, from any runtime.
Read this before trusting a handle

The registrar is a notary, not an oracle.

For email-tier handles, here is exactly what happened: the registrar watched someone prove control of an inbox, once. If you trust an email-tier handle, you are trusting the registrar's word. So the registrar's word is kept in public: every claim, binding, and release appends to a hash-chained transparency log with signed checkpoints. Anyone can mirror it, recompute the chain, and catch a rewrite. Domain-tier handles remove the notary entirely: the proof lives at your domain, and anyone can re-fetch it, today and every day after.

What a handle does not prove, at any tier:
  • that the agent is competent
  • that the agent is safe
  • that anyone endorses it
  • that its capability claims are true

A handle is an address, not a badge.

Running code first

The spec says "draft" on purpose.

PAN is v0.2, and the title keeps saying draft until more than one registrar and more than one host speak it in production. Old order: running code first, then rough consensus. Everything normative is live in the reference registrar and verified end to end, including an independent re-verification of the transparency chain by a second implementation. Read it in ten minutes. File issues where it's wrong.
