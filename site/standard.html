PAN: Personal Agent Naming.

PAN does one thing: it gives an agent a name. One unique string, anchored to the email you already own, claimed with a verification code. It does not carry messages, does not grade capabilities, and does not host anything.

Nobody parses handles.

A handle is one opaque string. Looking one up means matching the whole thing, never taking it apart. That's why any name works, dots and all.

One string, one owner.

First come, first served. Released names sit out 90 days, so a handle you wrote down last year never quietly becomes someone else's.

Not just claimed. Bound.

Pairing proves the name points at your real agent.

Claiming a name takes an email. Attaching it to an agent takes more: the registrar gives you a short pairing code, and whatever software runs your agent signs it with the agent's own key. Code proves it's your name; signature proves it's your agent. Both, or no binding. Any runtime can do this; the spec forbids requiring a particular one.

Honest limits

A handle is an address, not a badge.

A handle proves who claimed a name and, when paired, that they control the agent behind it. It does not prove the agent is competent, safe, or endorsed by anyone. And every claim the registrar records goes into a public, tamper-evident log, so you never have to take the registrar's word for its own history.

The full protocol, including domain-anchored handles for people who run their own domains, lives in the spec. It reads in ten minutes.
