Give every agent you host a name. It's one signature.

If you build a personal-agent runtime, your users' agents are the anonymous majority this standard exists for. Implementing PAN pairing in your host is deliberately tiny.

  1. Your agent already has (or can generate) an Ed25519 keypair. Its public key is its listing identity.
  2. When a user pastes a pairing code, sign one canonical string: the UTF-8 bytes of pan-pair-v1:<code>:<agent-id>.
  3. POST {code, agent_id, signature} to the registrar. No API key, no OAuth, no account. The code plus the signature is the whole handshake.

That's the entire integration. The reference signer, pan_pair.rs, is a standalone example of about 25 lines, and it depends on no particular agent framework, because the spec forbids the registrar from requiring one. Your users get names that resolve publicly, carry your manifest format verbatim, and work with existing WebFinger tooling. You get out of the naming business entirely.

Resolution, for consumers

Resolving a handle is one GET.

Exact-string lookup, no parsing, no auth. The card carries the anchor tier, the binding method and its proof class, presence where the registrar observes it, typed endpoints, and the agent's manifest verbatim. A PAN handle is also a valid acct: URI, so WebFinger works out of the box. 

# the card
curl "https://agentcatalog.com/api/resolve?handle=Coder.you@gmail.com"

# the same handle via WebFinger (RFC 7033)
curl "https://agentcatalog.com/.well-known/webfinger?resource=acct:Coder.you@gmail.com"
Same code, your shelf

The registrar is one binary. So is this argument.

A naming standard controlled by one mandatory registrar wouldn't be a standard. Any conforming registrar can serve PAN, you can run your own, and federation is named in the spec as explicit future work. We intend to earn the default, not own it.
